# Pitfalls Research

**Domain:** Personal Search / Local Vector RAG
**Researched:** 2026-02-05
**Confidence:** HIGH

## Critical Pitfalls

### Pitfall 1: The "DOM Drift" Death Spiral

**What goes wrong:**
Scrapers built for specific sites (Slack, Discord, Jira) break instantly when the target site pushes a UI update. The search index becomes stale, and the user blames the tool.

**Why it happens:**
Developers rely on brittle CSS selectors (e.g., `div.class-xyz > span:nth-child(2)`) that are often auto-generated by frameworks like React or styled-components, meaning they change with every build.

**How to avoid:**
*   **Semantic Selectors:** Anchor extraction to text content or ARIA labels (e.g., `[aria-label="Message body"]`) rather than structural classes.
*   **Visual-Spatial Heuristics:** Identify elements by their relationship to stable anchors (e.g., "the text block immediately below the timestamp").
*   **Local API Usage:** Prefer local APIs (e.g., Chrome Accessibility Tree) over raw DOM parsing where possible.

**Warning signs:**
*   Scrapers failing on specific sites after a weekend.
*   Selectors looking like `.css-1r345` or `#id-9932`.

**Phase to address:**
**Phase 1 (MVP)** - Establish a "resilient selector" policy immediately.

### Pitfall 2: Chrome Store "Privacy" Rejection

**What goes wrong:**
The extension is fully built but rejected by the Chrome Web Store because it requests broad permissions (e.g., `<all_urls>`, `tabs`, `history`) to index content, which triggers heightened review and rejection for "accessing local user data without adequate justification."

**Why it happens:**
Developers treat the browser extension as a "god mode" tool during local dev, ignoring the strict review policies regarding "Permission Creep" and handling of user data.

**How to avoid:**
*   **Optional Permissions:** Request site access only when the user explicitly adds that site to WIMS.
*   **Local-Only Guarantee:** Ensure *zero* data leaves the machine. If using an external LLM, make it opt-in with clear warnings.
*   **Justification:** Prepare the store listing justification text *before* writing the manifest.

**Warning signs:**
*   `manifest.json` permissions list growing longer than the feature list.
*   Collecting data "just in case" it's needed later.

**Phase to address:**
**Phase 2 (Extension Hardening)** - Audit permissions before the first internal release.

---

### Pitfall 3: Vector Index Bloat

**What goes wrong:**
The local vector database consumes gigabytes of disk space and RAM, slowing down the user's entire machine. A 5M vector index can easily eat 30GB+ if not optimized.

**Why it happens:**
*   **Over-indexing:** Storing full HTML or massive chunks instead of concise embeddings.
*   **Uncompressed Vectors:** Using high-dimensional float32 vectors for everything.
*   **No Garbage Collection:** Old, irrelevant data (e.g., deleted Slack messages) remains in the index forever.

**How to avoid:**
*   **Quantization:** Use int8 or binary quantization for vectors (reduces size by 4x-32x with minimal accuracy loss).
*   **Rolling Windows:** Implement a default "Time to Live" (TTL) for transient content (e.g., chat logs > 90 days).
*   **Hybrid Storage:** Store vectors in memory/fast DB, but raw content in slower, compressed storage (blob storage).

**Warning signs:**
*   `node_modules` or database folder size exceeds 1GB during testing with small datasets.
*   Laptop fans spinning up when the search tool is "idle" (likely re-indexing).

**Phase to address:**
**Phase 3 (Backend Optimization)** - Implement quantization and retention policies.

## Technical Debt Patterns

| Shortcut | Immediate Benefit | Long-term Cost | When Acceptable |
|----------|-------------------|----------------|-----------------|
| **CSS Class Selectors** | Fast to inspect and copy-paste. | Breaks on every site update (Weekly maintenance hell). | **Never** for reliable scrapers. |
| **In-Memory Vector DB** | Zero setup, super fast initial dev. | Crashes browser/process when data exceeds RAM. | **MVP / Prototype only**. |
| **"Universal" Scraper** | One script to rule them all (`document.body.innerText`). | Low quality noise, indexing navbars and footers. | **MVP only** for broad coverage. |

## Integration Gotchas

| Integration | Common Mistake | Correct Approach |
|-------------|----------------|------------------|
| **Single Page Apps (SPAs)** | Scraping `window.onload` but content loads via AJAX later. | Use `MutationObserver` to detect when content settles. |
| **Auth Walls** | Assuming cookies persist forever. | Detect auth failures and prompt user to re-login via the extension popup. |
| **Vector DB (WASM)** | Running heavy vector math on the main UI thread. | Offload all indexing/search to a **Web Worker** or Service Worker. |

## Performance Traps

| Trap | Symptoms | Prevention | When It Breaks |
|------|----------|------------|----------------|
| **Main Thread Indexing** | Browser UI freezes while typing or scrolling. | Web Workers for all CPU-intensive tasks. | > 100 documents. |
| **Naive Re-indexing** | Re-processing the entire history on every page load. | Checksums/Hashes of content before re-embedding. | > 1000 documents. |
| **Large Context Windows** | RAG fails because "prompt too long" or high latency. | Reranking step to select only top-k *relevant* chunks. | > 10 search results. |

## Security Mistakes

| Mistake | Risk | Prevention |
|---------|------|------------|
| **Indexing Secrets** | Searching for "API Key" returns your actual .env keys. | exclude rules for `.env`, `password`, `key` patterns by default. |
| **Cross-Context Leak** | "Work" search results showing up in "Personal" queries (or vice versa). | Strict metadata tagging (source: work/personal) and query-time filtering. |
| **Extension Injection** | Malicious site reading your local index via the extension. | `externally_connectable` set to `none` or strictly whitelisted domains. |

## UX Pitfalls

| Pitfall | User Impact | Better Approach |
|---------|-------------|-----------------|
| **The "Empty State" Void** | User searches, gets 0 results, assumes tool is broken. | "Did you mean...?" or "We've indexed X sites, check settings." |
| **Hallucinated Answers** | LLM confidently makes up facts about your data. | strict grounding: "I couldn't find that in your documents" fallback. |
| **Silent Failures** | Scraper breaks, user doesn't know until they need data. | Health dashboard: "Slack connection lost 2 days ago." |

## "Looks Done But Isn't" Checklist

- [ ] **Auth Refresh:** Handles session timeouts on indexed sites automatically?
- [ ] **Incognito Mode:** Does it accidentally index Incognito windows? (Should not).
- [ ] **Deduplication:** Does visiting `site.com` and `site.com/` result in two index entries?
- [ ] **Rate Limiting:** Does the scraper accidentally DDOS the user's own cloud tools (e.g., Notion API limits)?

## Recovery Strategies

| Pitfall | Recovery Cost | Recovery Steps |
|---------|---------------|----------------|
| **Corrupt Index** | MEDIUM | Button to "Nuke & Re-index" (user loses history but fixes tool). |
| **Banned IP** | HIGH | User is IP banned from a site due to aggressive scraping. **Throttle requests** immediately. |
| **Leaked PII** | HIGH | "Forget text" feature to delete specific vectors/chunks from index. |
