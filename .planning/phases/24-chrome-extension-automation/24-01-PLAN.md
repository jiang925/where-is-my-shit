<task type="auto">
  <name>Task 2: Add extension publishing job to GitHub Actions</name>
  <files>.github/workflows/release.yml</files>
  <action>
Extend .github/workflows/release.yml with new publish-extension job that runs after version-sync.

Job structure:
- Job name: publish-extension
- Depends on: version-sync
- Runs on: ubuntu-latest
- Conditional: only run on tag pushes (if: github.ref_type == 'tag')

Steps:
1. Checkout code (actions/checkout@v4)
2. Set up Node.js 20 for extension build (actions/setup-node@v4 with cache: npm)
3. Install extension dependencies (cd extension && npm ci)
4. Build extension (cd extension && npm run build)
5. Validate manifest.json exists and is valid JSON
6. Validate manifest.json version matches tag (extract from github.ref_name, compare with jq -r '.version' extension/dist/manifest.json)
7. Validate required manifest fields (name, version, manifest_version, description, icons, permissions)
8. Create extension zip (cd extension/dist && zip -r ../../extension.zip .)
9. Verify zip contains manifest.json (zipinfo extension.zip manifest.json)
10. Upload to Chrome Web Store using chrome-webstore-upload-cli:
    ```bash
    npx chrome-webstore-upload-cli upload \
      --extension-id ${{ secrets.CHROME_EXTENSION_ID }} \
      --client-id ${{ secrets.CHROME_CLIENT_ID }} \
      --client-secret ${{ secrets.CHROME_CLIENT_SECRET }} \
      --refresh-token ${{ secrets.CHROME_REFRESH_TOKEN }} \
      --source extension.zip \
      --auto-publish
    ```
11. Display success message with extension ID and version

Error handling:
- Set bash script to exit on first error (set -e)
- Each validation step should exit with code 1 on failure
- Use clear error messages for each validation failure
- Fail workflow if any step fails (don't proceed to upload if validation fails)

Secrets required (document in summary):
- CHROME_EXTENSION_ID: Extension ID from Chrome Web Store Developer Dashboard (32-char string)
- CHROME_CLIENT_ID: OAuth client ID from Google Cloud Console
- CHROME_CLIENT_SECRET: OAuth client secret
- CHROME_REFRESH_TOKEN: Refresh token obtained via chrome-webstore-upload-cli init

Rationale: Automated publishing reduces manual toil, ensures version consistency, and provides audit trail via GitHub Actions logs. Validation before upload prevents wasted time uploading invalid extensions.

Note: First manual submission required to obtain extension ID and set up Web Store listing. After first submission and credentials configured in GitHub Secrets, all subsequent updates are automated.

## Implementation Notes

**GitHub Secrets Security:**
- Secrets are encrypted at rest
- Only exposed to workflow runs
- Never logged (GitHub automatically masks secret values)
- Only repository maintainers can view/edit

**Chrome Web Store Review:**
- All submissions reviewed (automated or manual upload)
- Review criteria: permissions match functionality, no malicious code, privacy compliance
- Review time typically 24 hours, may be longer during high volume
- Rejections require manual intervention (fix issue, re-tag, re-submit)

**Rollback Strategy:**
- If bad version published, manual rollback via Developer Dashboard
- Unpublish bad version, push new tag with fix
- Automated workflow publishes rollback version
- Alternative: Keep previous version zip, manually upload for faster rollback

**Future Enhancements (Deferred):**
- Beta channel publishing (--trusted-testers flag)
- Automated screenshot generation for store listing
- Multi-language privacy policy (currently English only)
- Custom domain for privacy policy (currently using github.io)

- Manual first submission (prerequisite, cannot automate initial listing)
